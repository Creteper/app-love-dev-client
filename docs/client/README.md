# Client — 客户端

## 为什么客户端接口独立成端

「爱开发」的客户端是甲方用户的专属入口，它的业务流程和另外三端有本质区别：

- **AI 对话驱动**：客户不需要知道程序员等级、不需要看接单列表、不需要管财务图表。他唯一需要做的就是"向 AI 描述需求 → 确认 → 等交付"。客户端接口围绕这个流程设计。
- **钱包语义不同**：客户端钱包是「充值 + 托管」，和开发者端「提现 + 收款」是完全相反的资金流向。合并编写会导致变量命名和文案冲突。
- **群聊的视角不同**：客户端在项目群里作为甲方，看到的是进度报告、验收确认；开发者看到的是进度上传、请求验收。虽然调用的是同一个 WebSocket 通道，但 REST 接口中历史消息的过滤条件和操作权限不同。

**独立为 `client/` 的好处**：

- **语义清晰**：看 `client/api.md` 就完整理解甲方能做什么，不会和乙方（开发者）的接口混在一起。
- **前端对齐快**：客户端前端开发人员只需要读 `auth/` + `shared/` + `client/` 三个文件，不需要跳过开发者和后台的内容。
- **权限边界明确**：每个接口用 `权限：client` 标注，后端鉴权中间件可以直接映射。

**潜在缺点**：

- 和开发者端存在少量结构相似的接口（如查询项目详情 `GET /projects/:id`），写了两份会导致维护时有"改一处忘另一处"的风险。
- 客户端和开发者端之间的消息接口实际共享同一个后端服务，但文档里看起来像两套独立系统。

> **应对策略**：相似接口在文档中互相引用（"开发者端对应接口见 `../developer/api.md#xxx`"），代码层面后端应该统一路由前缀来做鉴权，而不是写两套 handler。
