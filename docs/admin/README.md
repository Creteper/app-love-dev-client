# Admin — 后台管理端

## 为什么管理端接口独立成端

后台管理端是整个平台的「上帝视角」，拥有管理员（`admin`）和客服（`cs`）两种角色。和客户/开发者端相比：

- **角色内部分层**：管理员拥有全部权限（用户管理、财务管理、全平台项目视图），客服只能看到自己被分配的项目和待对接客户。这种「同端不同权」的设计在代码层面通过 `role` 字段 + 接口级权限标注实现。
- **数据聚合需求**：财务图表（月度收入趋势、已结算/未结算堆叠）、项目状态分布、开发者等级分布——这些都是跨用户、跨项目的聚合查询，客户端和开发者端完全不需要。
- **状态机管理**：管理员/客服可以直接操作项目状态流转（冻结/解冻/分配开发者），这些是平台治理行为，不是用户自助操作。

**独立为 `admin/` 的好处**：

- **权限标注一目了然**：每个接口标注 `权限：admin` 或 `权限：admin | cs`，后端鉴权中间件直接映射，不会出现「客户端误调管理接口」的安全漏洞。
- **聚合查询集中管理**：Dashboard 图表所需的聚合 SQL / Redis 缓存逻辑只在 admin 文档中描述，避免污染客户/开发者端的简洁接口列表。
- **客服专属流程**：待对接客户（`/pending`）是客服的专属工作区——AI 生成需求初稿后由客服确认并发布，这个流程在客户和开发者端完全不可见。

**潜在缺点**：

- 管理员和客服共享同一个路由前缀，但权限差异需要通过每个接口的 `权限` 标注来区分。如果标注遗漏，可能导致权限泄露。
- 随着平台规模扩大，财务和用户管理可能膨胀为独立的微服务，届时 `admin/api.md` 会过长。

> **应对策略**：接口权限标注务必修到每一个端点。未来如需拆分，按功能域（`finance.md`、`users.md`、`projects.md`）分开，但当前 Demo 阶段统一管理更简洁。
