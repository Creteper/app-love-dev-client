# WebSocket — 即时通讯协议

## 为什么即时通讯单独出协议文档而不是写在各端的 REST 文档里

「爱开发」的项目群聊是平台的核心功能——客户、客服、开发者在同一个群里协调进度、上传截图、验收里程碑。这个功能**不能用 REST 实现**：

- **实时性**：消息需要在群内所有在线成员的屏幕上立刻出现。REST 轮询的延迟和带宽开销在产品层面不可接受。
- **双向推送**：进度报告、验收通知、AI 审核拦截——这些是服务端主动推送的事件，REST 只能客户端拉取。
- **消息类型多样**：不只是文本——图片、进度报告、交付物文件列表、需求规格卡片——每种类型对应不同的前端渲染组件。REST 风格下这些多态消息的序列化协议会很难一致。

**独立为 `websocket/` 的好处**：

- **协议独立于业务路由**：WebSocket 的连接握手、心跳、重连策略是传输层问题，和「客户端有哪些接口」「开发者有哪些接口」无关。
- **消息格式定义一次、三端复用**：客户、开发者、客服看到的群聊界面不同，但它们发出的消息格式完全相同。统一协议文档避免了三端文档各自定义一套不兼容的 JSON。
- **AI 审核拦截是协议级行为**：当消息被 AI 判定为引流/敏感内容时，服务端需要立刻告诉发送方「你的消息被拦截了」，同时**不广播给群内其他成员**。这种「发送方收到 reject、群内无感知」的行为是协议层面的，不适合写在 REST 的「获取历史消息」接口里。

**潜在缺点**：

- WebSocket 没有 REST 那样的 HTTP 状态码（200/400/422），错误反馈依赖消息体内的 `type: "error"`，调试时不如 REST 直观。
- 断线重连时的消息补发策略、离线消息的存储和拉取顺序，需要和 REST 的 [客户端 API](../client/api.md#63-获取历史消息) 协同定义，但当前文档中两者的交互只做了简要说明。

> **应对策略**：WebSocket 负责实时消息推送，REST 负责历史消息拉取和群信息查询。两者通过 `messageId` 作为一致性锚点——断线重连后客户端从本地最后一条 `messageId` 开始拉 REST 历史消息补齐缺口。
