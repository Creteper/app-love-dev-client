# Auth — 鉴权模块

## 为什么鉴权独立为一个模块而不是分散在三端各写一份

「爱开发」有客户、开发者、管理员/客服三种身份，看似是三套独立的登录注册流程，但它们的鉴权骨架只有 **两种模式**：

- **客户 & 开发者**：前端注册 + 登录，后端签发 JWT
- **管理员 / 客服**：后台账户预分配，登录后按 `role` 字段区分权限

如果把鉴权拆到各端文档中：

- **客户端的注册和开发者的注册 90% 是一样的**（邮箱 + 密码 + 验证码），唯一区别是开发者注册时需要选技能字段。拆开写会导致相同逻辑写两遍，改一处漏一处。
- **Token 刷新逻辑三条完全一样**——`POST /refresh-token` 的请求/响应格式、401 自动重试策略、并发刷新去重，这些是三端共享的基础设施。
- **后台管理的登录本质上同样是 JWT 签发**，只是多了一个 `role` 字段来区分管理端内部的管理员 vs 客服。

**独立为 `auth/` 的好处**：

- **减少歧义**：所有端的认证行为在这里定义一次，不需要跨文档对齐。
- **安全审计集中**：JWT 过期策略、刷新去重机制、密码哈希方案——这些安全关键逻辑在一处声明，审核只需看这一个文件。
- **扩展性**：未来如果加入「扫码登录」「第三方 OAuth」，只需要在 `auth/` 追加，三端都能引用。

**潜在缺点**：

- 三端认证接口的路径前缀不同（`/client`、`/developer`、`/admin`），初学者可能困惑"到底该调哪个"。文档需要在开头明确说明路由区分方式。
- 如果未来三端的鉴权逻辑开始显著分化（比如客户端用短信验证码、开发者用 OAuth、后台用 LDAP），这个文件会膨胀。届时需要拆分子文件（`auth/client.md`、`auth/developer.md`、`auth/admin.md`）——但目前阶段统一管理更优。

> **判断标准**：只要三端共享 Token 签发、刷新、撤销的同一套后端服务，就应该维持一个 `auth/api.md`。只有当不同端接入完全不同的认证服务时，才考虑拆分。
