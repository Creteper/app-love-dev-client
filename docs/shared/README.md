# Shared — 通用规范

## 为什么把通用规范独立为一个模块

「爱开发」是一个四端平台（客户 / 开发者 / 后台管理），各端虽然面向不同角色，但在接口层面存在大量共享约定——错误码、分页格式、时间格式、认证传递方式、文件上传策略、跨域策略等。

如果把这些内容分别写在每一端的文档里：

- **重复劳动**：同一份错误码表要在四份文档中维护，任何增改都会导致四处不同步。
- **不一致风险**：不同端的开发者可能对这层约定的理解出现偏差，最终导致前后端对接成本上升。
- **维护成本高**：新同事接手某一端时，不知道该端文档里的「通用约定」和其他端是否一致，得逐个对比确认。

**独立为 `shared/` 的好处**：

- **单一事实来源（Single Source of Truth）**：所有端引用同一份 `common.md`，不会出现版本漂移。
- **新人友好**：先看 `shared/` 再切入任一端，公共规则一目了然。
- **抽查方便**：Code Review 时只需确认 PR 是否遵循 `shared/` 即可，不需跨端检查。

**潜在缺点**：

- 引用链多一层：端内文档需要显式指向 `../shared/common.md`，易被忽略。
- 如果不小心让某个端"绕开"共享规范写了自己的约定，反而更难发现不一致（因为 reviewer 会默认「共享规范里管好了」）。

> **应对策略**：端内文档不重复撰写通用约定，而是在开头显式引用 `../shared/common.md`。后续 CI 可加入链接有效性检查。
