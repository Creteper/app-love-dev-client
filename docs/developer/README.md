# Developer — 程序员端

## 为什么程序员端接口独立成端

程序员端是乙方用户的入口，它包含「接单大厅」和「开发者工作台」两个子端，但后端服务是同一套。程序员端的核心差异在于：

- **资金流向相反**：程序员端是「提现 + 收款」，客户端是「充值 + 托管」。即使字段名相似（`available` / `frozen`），语义完全不同——一个是可提现金额，一个是可用余额。
- **等级体系**：A/B/C 三级决定了能接什么难度的单。等级计算依赖完单数和评分，这是程序员端独有的业务逻辑。
- **每日进度上报**：项目要求开发者每天至少上传一次进度（截图/现场照片），这是平台透明交付的核心机制。这个流程在客户端和管理端都没有对应的写操作。
- **抢单机制**：接单大厅的「先到先得」模式需要后端做并发控制（乐观锁/分布式锁），这种写操作的并发语义在客户端不存在。

**独立为 `developer/` 的好处**：

- **接口权限天然隔离**：`role=developer` 的 JWT 只能访问 `developer/*` 前缀的接口，不会误调客户端的充值或管理端的封号操作。
- **等级和技能标签**：开发者独有的字段（`level`、`skills`、`rating`、`finishedOrders`）全部集中在一处，不需要在通用模型里做「某些端用某些字段」的条件判断。
- **接单并发语义明确**：抢单接口（`POST /orders/:id/accept`）需要在文档中显式标注「并发安全」要求，这是开发者端独有的技术约束。

**潜在缺点**：

- 和客户端的项目详情接口有 70% 的字段重叠，维护时需要两边同步。
- 开发者端内部的「接单大厅」和「工作台」没有进一步拆分为子模块，未来如果接单大厅膨胀为独立 App，当前 API 文档需要重新拆分。

> **应对策略**：客户端和开发者端共享的实体（项目、消息）在后端应使用统一的 Service 层和 DTO，接口文档中的差异仅在于哪些字段对哪一端可见、哪些操作被允许。
